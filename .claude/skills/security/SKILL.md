# SECURITY — 安全守卫（灵感来自 Everything Claude Code / AgentShield）

一人公司被黑一次可能就完蛋了。上线前必须过一遍安全检查。

## 什么时候触发

| 场景 | 自动运行 |
|------|---------|
| `/ship` 发布流程中 | ✅ 自动包含 |
| 用户说 `/security` | ✅ 手动触发 |
| 涉及用户数据的功能 | ✅ 建议运行 |

## 安全检查清单（新手版，只查最致命的）

### 1. 密钥泄露检查 🔑（最常见的致命错误）
```
扫描所有文件，检查是否包含：
□ API密钥硬编码在代码里？（应该在.env中）
□ .env文件被提交到Git了？（.gitignore必须包含.env）
□ 前端代码暴露了密钥？（浏览器能看到的都不安全）
□ 日志里打印了敏感信息？（console.log密钥/密码）
□ 配置文件里有真实密码？（应该用环境变量）

扫描关键词：
- API_KEY, SECRET, PASSWORD, TOKEN, PRIVATE_KEY
- sk-, pk-, ghp_, gho_
- Bearer, Authorization
```

### 2. 输入验证检查 🛡️（防止被攻击）
```
□ 用户输入有过滤吗？（防XSS攻击）
□ 数据库查询用参数化了吗？（防SQL注入）
□ 文件上传有限制吗？（类型、大小）
□ URL参数有验证吗？
□ 表单有CSRF保护吗？
```

### 3. 权限检查 🔒
```
□ API接口有身份验证吗？（不登录能调用吗？）
□ 用户只能访问自己的数据吗？（不能看别人的）
□ 管理功能有权限控制吗？
□ 敏感操作有二次确认吗？（删除、支付）
```

### 4. 依赖安全 📦
```
□ npm/pip依赖有已知漏洞吗？
  → npm audit / pip-audit
□ 依赖版本是否太旧？
□ 有没有不必要的依赖？（减少攻击面）
```

### 5. 部署安全 🌐
```
□ HTTPS已启用？（Vercel/Railway默认有）
□ 错误信息不会暴露系统细节？（生产环境关debug模式）
□ CORS配置正确？（不要用 * 允许所有来源）
□ HTTP安全头设置了？（X-Frame-Options, CSP等）
```

## 安全报告格式

```markdown
## 安全检查报告
- 检查日期：YYYY-MM-DD
- 检查范围：[全项目 / 指定模块]

### 🔴 严重问题（必须立即修复）
- 问题描述 → 文件位置 → 修复方案

### 🟡 中等风险（建议修复）
- 问题描述 → 文件位置 → 修复方案

### 🟢 低风险/建议
- 改进建议

### ✅ 已通过的检查
- 列出所有通过的项目

### 结论
[安全 / 需要修复后再上线]
```

## 快速修复指南

| 问题 | 一句话修复 |
|------|-----------|
| 密钥在代码里 | 移到 `.env`，代码中用 `process.env.KEY` |
| .env被提交了 | 加入 `.gitignore`，从Git历史中删除 |
| 没有输入验证 | 用框架自带的验证（如 Zod, Pydantic） |
| SQL注入风险 | 用 ORM 或参数化查询，不要拼接SQL |
| 没有HTTPS | 用 Vercel/Railway 部署（自动HTTPS） |

## 原则
- **默认不信任** — 所有用户输入都当作恶意的
- **最小权限** — 只给必要的权限，不多给
- **失败安全** — 出错时拒绝访问，不是放行
- **新手也能懂** — 每个问题都给出具体的修复步骤
