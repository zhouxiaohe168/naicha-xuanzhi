# WORKFLOW — 标准化工作流编排器

把所有工作流程标准化，像流水线一样运行，不漏步骤。

## 当用户说 /workflow 时

### Step 1: 识别工作流类型

| 类型 | 触发条件 | 标准流程 |
|------|---------|----------|
| **新项目** | "我要做一个新XX" | PRD → Checklist → Build → Test → Deploy |
| **新功能** | "帮我加一个XX功能" | 需求确认 → 设计 → 开发 → 测试 → 合并 |
| **Bug修复** | "XX出问题了" | 复现 → 定位 → 修复 → 测试 → 部署 |
| **内容发布** | "帮我写XX" | 创作 → 审核 → 排版 → 发布 → 数据跟踪 |
| **每日运营** | "今天该做什么" | 检查系统 → 处理优先事项 → 内容 → 复盘 |

---

## 标准工作流定义

### 工作流 A：新项目（完整流程）

```
阶段0 — 想法验证
│  /idea → 评估想法
│  /compete → 竞品分析
│  决策：做 or 不做？
│
阶段1 — 需求定义 ⚠️ 不可跳过
│  /prd → 反向提问 → 生成需求书
│  /mermaid → 思维导图确认逻辑
│  用户确认："需求OK"
│
阶段2 — 准备阶段 ⚠️ 不可跳过
│  /checklist → 生成准备清单
│  准备所有 API 密钥、账号、素材
│  用户确认："准备齐全"
│
阶段3 — 技术设计
│  选择技术方案
│  设计数据库结构
│  画架构图（/mermaid）
│  用户确认："方案OK"
│
阶段4 — 开发
│  /build → 创建项目骨架
│  子智能体并行开发
│  每完成一个模块 → 汇报进度
│  GitHub 提交代码
│
阶段5 — 测试
│  Playwright 自动化测试
│  手机/电脑多设备测试
│  修复所有问题
│  用户验收
│
阶段6 — 部署上线
│  部署到生产环境
│  配置域名
│  线上验证
│  用户确认："上线成功"
│
阶段7 — 推广运营
│  /content → 写推广内容
│  /pitch → 准备介绍话术
│  发布到社交媒体
│
阶段8 — 复盘迭代
│  /review → 周复盘
│  收集用户反馈
│  规划下一版迭代
```

### 工作流 B：新功能

```
1. 需求确认
│  "你想要什么功能？为什么？"
│  评估：是否在MVP范围内？
│  更新思维导图
│
2. 影响分析
│  这个功能会影响哪些现有代码？
│  需要改数据库吗？
│  需要新的API吗？
│
3. 开发
│  创建 Git 分支
│  写代码
│  写测试
│
4. 测试
│  自动化测试
│  人工验收
│
5. 合并部署
│  合并到主分支
│  部署更新
│  线上验证
```

### 工作流 C：Bug 修复

```
1. 复现问题
│  "具体什么情况下出问题？"
│  能复现 → 进入定位
│  不能复现 → 收集更多信息
│
2. 定位原因
│  查看日志/报错信息
│  Explore 子智能体定位代码
│  确认根本原因
│
3. 修复
│  写修复代码
│  确保不引入新Bug
│
4. 验证
│  Playwright 测试修复效果
│  回归测试（原有功能不受影响）
│
5. 部署
│  提交 + 部署 + 验证
```

### 工作流 D：每日运营

```
早上：
├── 检查系统运行状态
├── 查看用户反馈/消息
└── 确定今日最重要的1件事

上午（深度工作）：
├── 处理最重要的事
└── 开发/修复/优化

下午：
├── 内容创作（/content）
├── 社区互动
└── 学习新东西（/learn）

晚上：
└── /review 日复盘（5分钟）
```

## 工作流自动化规则

### 自动触发（我会自动执行）

| 当你说... | 我自动执行 |
|----------|-----------|
| "我想做一个新XX" | /prd → /checklist → /build 完整流程 |
| "帮我加一个XX功能" | 先确认需求 → 评估影响 → 再开发 |
| "XX坏了/出问题了" | 先复现 → 定位 → 再修复 |
| "今天做什么" | /opc 每日启动仪式 |
| "这周怎么样" | /review 周复盘 |

### 流程守卫（我会阻止你跳步骤）

```
⚠️ 如果你说"直接开始写代码"但还没有PRD：
   → 我会说："兄弟，先花10分钟用 /prd 把需求理清楚，磨刀不误砍柴工。"

⚠️ 如果你说"上线吧"但还没测试：
   → 我会说："等一下，让我先用Playwright跑一遍测试。"

⚠️ 如果你要加功能但已经超出MVP范围：
   → 我会说："这个放到第二版吧？先把核心做好上线。"
```

## 原则
- **流程是护栏不是枷锁** — 简单项目可以精简步骤
- **每一步都要有交付物** — PRD有文档、开发有代码、测试有报告
- **你确认了才往下走** — 关键节点必须等你拍板
