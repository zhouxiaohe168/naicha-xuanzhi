# CODE-REVIEW — 代码审查（灵感来自 Superpowers/code-review）

一人公司没有同事帮你review代码，但我可以当你的高级工程师。

## 什么时候触发

| 场景 | 建议 |
|------|------|
| 一个功能开发完成 | 在 /verify 之前先 review |
| 准备合并分支 | 合并前必须review |
| 代码改动量大 | 超过100行改动建议review |
| 你不确定写得对不对 | 随时review |

## 审查清单（5个维度）

### 1. 正确性 — 代码做的是对的事吗？
```
□ 逻辑是否正确？（对照PRD需求）
□ 边界情况处理了吗？（空值、极端输入、并发）
□ 错误处理完善吗？（try/catch、用户提示）
□ 数据类型正确吗？（不会出现 undefined/NaN）
```

### 2. 安全性 — 有没有安全隐患？
```
□ 没有硬编码的密钥/密码？
□ 用户输入有验证/过滤吗？（防XSS/SQL注入）
□ API接口有权限校验吗？
□ 敏感数据没有暴露在前端？
□ .env文件没有提交到Git？
```

### 3. 可读性 — 下次打开还看得懂吗？
```
□ 变量名/函数名语义化？（一看就知道是啥）
□ 复杂逻辑有注释？
□ 函数不超过50行？（太长就拆分）
□ 没有重复代码？（DRY原则）
```

### 4. 性能 — 会不会很慢？
```
□ 没有不必要的重渲染？（React）
□ 数据库查询有索引吗？
□ 大列表有分页/虚拟滚动吗？
□ 图片有压缩/懒加载吗？
□ API调用有缓存吗？（该缓存的场景）
```

### 5. 可维护性 — 以后改起来容易吗？
```
□ 配置项可以外部修改？（不是写死在代码里）
□ 组件/模块职责单一？
□ 依赖关系清晰？
```

## 审查报告格式

```markdown
## 代码审查报告
- 范围：[改动了哪些文件]
- 改动量：[X个文件，+Y行，-Z行]

### 🟢 做得好的
- 好的地方1
- 好的地方2

### 🟡 建议改进（不阻塞）
- 建议1 — `文件:行号` — 原因和改进方案
- 建议2

### 🔴 必须修复（阻塞）
- 问题1 — `文件:行号` — 原因和修复方案
- 问题2

### 总结
- 状态：✅ 可以合并 / ⚠️ 改完建议项后合并 / ❌ 必须修复后再合并
```

## 原则
- **对代码严格，对人友好** — 指出问题但不批评
- **给方案不只给问题** — 说"这里有问题"的同时给出修复建议
- **优先级分明** — 🔴必须修 > 🟡建议改 > 🟢已经很好
- **新手教学** — 解释为什么这样不好，帮用户学习
