# SHIP — 发布流水线（灵感来自 Superpowers/finishing-a-development-branch）

从"代码写完了"到"用户能用了"的最后一公里。

## 当用户说 /ship 或 "上线吧" 时

### 自动执行发布流水线

```
Step 1: 代码审查
│  /code-review → 确保代码质量
│  🔴 有必须修复的 → 修完再来
│  ✅ 通过 → 继续
│
Step 2: 安全检查
│  /security → 扫描密钥泄露、输入验证、权限问题
│  🔴 有严重问题 → 修完再来
│  ✅ 通过 → 继续
│
Step 3: 完成验证
│  /verify → 确保功能完整
│  ❌ 未通过 → 修完再来
│  ✅ 通过 → 继续
│
Step 4: Git 收尾
│  确保所有代码已提交
│  合并到主分支（如果在分支上开发）
│  打版本标签（v1.0, v1.1...）
│
Step 4: 部署
│  构建生产版本
│  部署到生产环境
│  Playwright 线上验证
│
Step 5: 发布确认
│  你打开线上网址确认
│  ✅ "没问题" → 继续
│  ❌ "有问题" → 回滚 + 排查
│
Step 6: 发布公告（可选）
│  /content → 生成发布推文/更新日志
│  你确认后发布
│
Step 7: 收尾
│  /handoff → 保存发布记录
│  更新 CHANGELOG.md
│  清理临时分支
```

### 发布检查清单

```
发布前必查：
□ 代码审查通过（/code-review ✅）
□ 功能验证通过（/verify ✅）
□ 所有代码已提交 Git
□ 环境变量已配置
□ 数据库迁移已执行（如果有）
□ 没有 console.log / 测试数据
□ .env 没有提交到仓库
□ HTTPS 配置正确
□ 移动端显示正常

发布后必查：
□ 线上能正常访问
□ 核心功能能用
□ 没有控制台报错
□ 性能可接受（< 3秒加载）
```

### 版本号规则（简单版）

```
v1.0.0 — 首次上线
v1.0.1 — 修Bug
v1.1.0 — 加新功能
v2.0.0 — 大改版
```

### 回滚方案

如果上线后发现严重问题：
```
1. 不要慌
2. 回滚到上一个版本（Git revert）
3. 用 /debug 排查问题
4. 修复后重新走 /ship 流程
```

## 原则
- **不急** — 多花10分钟检查，少花10小时救火
- **可回滚** — 永远保留回到上一版的能力
- **有记录** — 每次发布都写 CHANGELOG
- **庆祝** — 上线成功了记得给自己点个赞 🎉
